Reorganização da interface
Auditoria, três direções, matriz de ações e protótipo de alta fidelidade da direção recomendada — para validar posicionamento, hierarquia e usabilidade antes de tocar em produção.
1 · Auditoria curta da interface atual
Duplicação de ações
Pull/Push/Fetch/Commit existem em 3–4 sítios (header, barra inferior, sidebar, menus). Nenhum é claramente "o" sítio.
Alvos e texto pequenos
A 900×560, 43/53 controlos têm <32px numa dimensão; labels de 10–11px em ações funcionais; a caixa de commit sai do ecrã.
Descoberta escondida
Checkout por duplo clique e operações essenciais só no botão direito; de 56 elementos clicáveis no Histórico só 1 é alcançável por teclado.
Escala
A sidebar lista todas as branches sem pesquisa, pastas ou favoritas — não sobrevive a 200 branches nem a 500 ficheiros.
Hierarquia plana
Commit (a ação nº1) tem o mesmo peso visual que Tag (rara). Ações destrutivas ao lado das frequentes.
O que já está bem
Identidade da árvore, 4 temas com tokens sólidos, Ctrl+K funcional, informação Git credível. Preservar tudo isto.

3 · Direção recomendada: A, com o melhor de C
A — ações contextuais vence porque resolve as três dores medidas: duplicação (cada ação passa a ter UM local principal), densidade (sem barra inferior permanente sobra ~54px verticais a 900×560, o suficiente para a caixa de commit ficar visível) e aprendizagem (a ação aparece onde o objeto está: Commit na cópia de trabalho, Checkout na branch, Revert no commit). B mantém a causa raiz — uma barra global de ações contextuais — e a personalização adia o problema para o utilizador. C é o mais rápido para power users mas falha "compreensível para quem não conhece atalhos": descoberta por paleta pura mede-se mal com utilizadores intermédios. A proposta final é A com a paleta Ctrl+K de C como local alternativo universal e atalhos regraváveis.

7 · O que mudou
Mantido: header com sync, sidebar com navegação, Ctrl+K, temas, árvore, detalhe do commit.
Movido: Commit → cópia de trabalho (única ação primária); Branch/Tag/Revert → detalhe do commit; Checkout → botão explícito na row da branch; Definições → só header + Ctrl+K.
Combinado: toasts + notificações num só stack; Fetch/Pull/Push num grupo de sync com estado partilhado.
Removido: barra inferior permanente; barra de abas de repositórios (o seletor com abertos/registados/recentes preserva o contexto numa linha só); pesquisa de branches sempre visível (lupa, ou automática com muitas branches); entrada Definições da sidebar; "Saltar" do onboarding; duplo clique como único caminho de checkout.
Menu secundário ("…" e Ctrl+K): cherry-pick, rebase, reset, renomear, apagar branch, terminal.

9 · Acessibilidade
No produto real: button nativos + ARIA (menus com role="menu", aria-expanded/aria-haspopup, setas ↑↓, Escape fecha e devolve o foco, clique fora fecha) + Enter/Espaço (o protótipo usa tabindex só para a prova visual) · o grafo do histórico é hardcoded para os 8 commits de exemplo — a implementação gera lanes/curvas da topologia real · foco visível de 2px na cor do accent · botões 36px, rows importantes 40px, secundárias ≥32px · checkout por botão (não só duplo clique) · destrutivas com confirmação e Cancelar focado por omissão · prefers-reduced-motion desliga tudo · contraste AA nos labels (usar os --muted já corrigidos) · toasts com aria-live no produto real.
10 · Riscos a testar com utilizadores
1. Sem barra inferior, os utilizadores atuais encontram Merge/Stash/Tag? (medir tempo até à 1ª utilização via "…"/Ctrl+K)
2. Checkout: sempre visível na branch selecionada (como aqui); nas restantes por hover/foco — validar a descoberta
3. O detalhe em baixo a 900×560 é preferível a esconder o detalhe?
4. 200 branches: pesquisa + pastas chegam, ou é preciso um ecrã dedicado?
5. O grupo de sync no header lê-se como um só sistema?
6. Sem abas, quem alterna entre 2 repos a cada minuto sente atrito? (o seletor guarda o estado de cada repo aberto; medir a frequência de troca)
